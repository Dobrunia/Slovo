# WebSocket / LiveRail Flow

Этот файл описывает текущее состояние realtime-логики проекта `Slovo`.
Здесь зафиксировано именно то, что уже реализовано на текущий момент.

## Общая модель

- WebSocket-транспорт работает через `Socket.IO`.
- Realtime-контракты и runtime построены на `LiveRail`.
- Аутентификация сокета идет через session token из handshake.
- После подключения сервер вычисляет `userId` из сессии и строит realtime context.

## Какие realtime-каналы есть сейчас

### 1. Server structure channel

Канал: `server.structure`

Назначение:
- live-обновление названия сервера;
- live-обновление аватара сервера;
- live-обновление списка каналов;
- live-обновление порядка каналов.

Кто может подписаться:
- только участник конкретного сервера.

Ключ канала:
- `serverId`

### 2. Server presence channel

Канал: `server.presence`

Назначение:
- live-поток присутствия пользователей;
- кто сейчас находится в каком голосовом канале;
- вход в канал;
- выход из канала;
- переход между каналами.

Кто может подписаться:
- только участник конкретного сервера.

Ключ канала:
- `serverId`

### 3. Voice session channel

Канал: `voice.session`

Назначение:
- канал зарезервирован под channel-scoped voice/media-события.

Ключ канала:
- `serverId`
- `channelId`

Смысл:
- это уже подкомната внутри сервера;
- именно здесь модель рассчитана на то, что общение должно быть изолировано по конкретному каналу.

### 4. Voice signaling channel

Канал: `voice.signaling`

Назначение:
- канал зарезервирован под signaling для voice/media.

Ключ канала:
- `serverId`
- `channelId`

Смысл:
- signaling и media должны быть channel-scoped, а не server-wide.

### 5. User profile channel

Канал: `user.profile`

Назначение:
- события профиля конкретного пользователя;
- обновление списка серверов пользователя;
- forced disconnect для конкретного пользователя.

Кто может подписаться:
- только сам пользователь.

Ключ канала:
- `userId`

## Что считается комнатой сейчас

Сейчас есть два уровня realtime-изоляции:

### Уровень 1. Сервер

Сервер выступает как общая realtime-область видимости.

На этом уровне участники:
- видят структуру сервера;
- видят presence всех участников сервера;
- понимают, кто в каком канале находится.

То есть `server.structure` и `server.presence` работают на уровне всего сервера.

### Уровень 2. Канал

Канал выступает как отдельная подкомната внутри сервера.

Именно для него предусмотрены:
- `voice.session`
- `voice.signaling`

Логика здесь такая:
- внутри одного сервера можно видеть всех, кто где находится;
- но voice/media-общение должно быть только внутри конкретного канала;
- пользователь не должен слышать участников из другого канала того же сервера.

## Что уже работает прямо сейчас

Сейчас реализовано:

- runtime presence пользователя;
- пользователь одновременно может быть только в одном канале;
- команды:
  - `voice.join-channel`
  - `voice.leave-channel`
  - `voice.move-channel`
- событие:
  - `presence.updated`
- client UI:
  - счетчики участников у каналов;
  - список пользователей под каналом;
  - список пользователей в центральной области выбранного канала.

То есть на текущем этапе уже работает realtime-модель присутствия.

## Что пока еще не реализовано

Пока еще не реализован actual voice/media layer:

- нет `MediaSoup foundation`;
- нет transport/producers/consumers;
- нет реального аудио между участниками;
- нет реального screen share.

Поэтому сейчас изоляция по каналам уже заложена в websocket-модели и route keys,
но фактическое "слышать только свой канал" пока еще не доведено до media-уровня.

Иначе говоря:

- presence и live-state уже channel-aware;
- media isolation пока только подготовлена архитектурно, но еще не активна как audio flow.

## Как выглядит flow пользователя сейчас

### Вход в сервер

1. Клиент грузит initial snapshot сервера через GraphQL.
2. Клиент поднимает realtime-подписки:
   - `server.structure`
   - `server.presence`

### Вход в канал

1. Пользователь кликает по каналу.
2. Клиент отправляет LiveRail-команду `voice.join-channel`.
3. Сервер валидирует membership и наличие канала.
4. Сервер обновляет runtime presence registry.
5. Сервер эмитит `presence.updated` в `server.presence`.
6. Все клиенты этого сервера обновляют UI.

### Переход в другой канал

1. Клиент отправляет `voice.move-channel`.
2. Сервер меняет `channelId` в runtime presence.
3. Сервер эмитит `presence.updated`.
4. UI пересчитывает:
   - список под старым каналом;
   - список под новым каналом;
   - счетчики;
   - центральную область выбранного канала.

### Выход из канала

1. Клиент отправляет `voice.leave-channel`.
2. Сервер удаляет runtime presence пользователя.
3. Сервер эмитит `presence.updated`.
4. Пользователь исчезает из UI канала и счетчиков.
