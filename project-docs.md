# Документация проекта Slovo

## Обзор проекта

Slovo — это voice-first платформа для общения, построенная вокруг серверов и живых голосовых каналов. Она рассчитана на небольшие сообщества, команды и группы, которым нужно быстро собираться в общих пространствах, видеть, кто сейчас активен, мгновенно подключаться к разговору и показывать экран без text-first сценария.

Проект решает задачу легкой синхронной коммуникации. Вместо того чтобы строить опыт вокруг длинной истории сообщений, Slovo сочетает постоянную структуру сервера с live presence, голосовыми сессиями, приглашениями, модерацией и демонстрацией экрана. В результате получается система, где пользователь может найти сервер, зайти в канал, увидеть, кто находится внутри прямо сейчас, и координироваться в реальном времени.

## Пользовательские возможности

### Возможности гостя

- Открыть лендинг и перейти к регистрации или входу.
- Зарегистрировать новый аккаунт по email, username, password и display name.
- Войти в существующий аккаунт.
- Открыть invite-ссылку. Если пользователь еще не авторизован, приложение переводит его через auth-flow, а затем возвращает в сценарий вступления по приглашению.

### Возможности авторизованного пользователя

- Загружать текущий профиль и оставаться в системе через сохраненный session token.
- Видеть список серверов, в которых он состоит.
- Создавать собственные серверы. Текущий лимит продукта — до 2 серверов на одного owner.
- Открывать сервер и видеть его каналы и текущих участников.
- Искать публичные серверы и видеть рекомендованные популярные публичные серверы.
- Вступать в публичный сервер.
- Вступать в сервер по invite-ссылке.
- Редактировать профиль:
  - display name
  - avatar URL
- Настраивать персональные клиентские предпочтения:
  - включать или отключать интерфейсные звуки
  - выбирать устройство ввода звука

### Голосовые возможности и presence

- Видеть, кто сейчас находится в каждом голосовом канале.
- Заходить в голосовой канал.
- Переходить между каналами.
- Выходить из текущего канала.
- Видеть активных участников в выбранной области канала.
- Говорить и слышать других участников того же канала.
- Использовать self mute.
- Использовать self deafen.
- Слышать звуки входа и выхода из канала, если звуки включены.
- Запускать демонстрацию экрана.
- Останавливать демонстрацию экрана.
- Смотреть активную демонстрацию экрана других участников в том же канале.

### Возможности владельца сервера

- Менять название сервера.
- Менять аватар сервера.
- Получать текущую invite-ссылку сервера.
- Перегенерировать invite-ссылку.
- Создавать каналы. Текущий лимит продукта — до 5 каналов на сервер.
- Переименовывать каналы.
- Удалять каналы.
- Менять порядок каналов.
- Просматривать список участников для модерации.
- Исключать участника из сервера.
- Банить участника.
- Удалять сервер.

### Роли и права

- Сейчас в системе есть только 2 роли:
  - `OWNER`
  - `MEMBER`
- У сервера всегда ровно один owner.
- Владение сервером — это единственная административная роль в текущем продукте.
- Участники могут вступать в сервер и пользоваться голосовыми возможностями, но не могут управлять структурой сервера, invite-ссылкой или модерацией.

## Архитектура системы

### Структура репозитория

Репозиторий разделен на 3 основные части:

- `client/`: Vue-приложение
- `server/`: Node.js backend
- `shared/`: легковесные общие константы и имена

### Клиентская часть

Клиент — это Vue 3 приложение, организованное вокруг страниц, layout-ов, feature-модулей, composable-ов, GraphQL-клиентов, realtime-helper-ов и Pinia store-ов.

Основные ответственности клиента:

- роутинг и навигация с учетом авторизации
- инициализация сессии и состояние профиля
- список серверов и состояние выбранного сервера
- UI списка каналов и голосового присутствия
- realtime-подписки и отправка команд
- voice и screen share session handling через mediasoup-client
- owner UI для управления сервером, каналами, invite-ссылками и модерацией

Главный экран приложения собирается через authenticated layout, в который входят:

- верхний header со списком серверов и глобальными действиями
- список каналов сервера
- панель управления текущим пользователем
- центральная область канала и presence

### Серверная часть

Сервер — это TypeScript-приложение на Node.js с явным разделением между persistent API, realtime transport и media-layer.

Основные ответственности сервера:

- подъем HTTP, GraphQL, realtime и media-слоев
- аутентификация пользователей и резолв session-backed identity
- хранение постоянного состояния в MySQL через Prisma
- обработка GraphQL queries и mutations через StrictQL + Yoga
- выполнение realtime-команд и рассылка live-обновлений через LiveRail
- поддержка runtime-only presence state
- bridge между signaling и MediaSoup

### Основные серверные слои

- `config/`: runtime-константы и дефолты
- `data/`: подключение Prisma и управление жизненным циклом data layer
- `graphql/`: StrictQL operations, context и GraphQL server wiring
- `auth/`: auth policies, session resolution и current-user helpers
- `realtime/`: contracts, foundations, команды, события, disconnect-handling, presence и transport helpers
- `media/`: MediaSoup foundation и signaling bridge
- `server/`: доменные helper-ы для серверов, каналов и модерации
- `shutdown/`: идемпотентный graceful shutdown coordinator

### Общий слой

`shared/` намеренно остается легковесным. В нем лежат cross-package константы и realtime names/definitions, но package-specific runtime code остается внутри `client/` и `server/`.

Это важно, потому что:

- клиент забирает файлы из `shared/` в Vite bundle
- сервер включает файлы из `shared/` в TypeScript build
- dependency-heavy runtime schemas и framework bindings остаются package-local

## Технический flow / Как это работает

### 1. Аутентификация и сессии

Аутентификация начинается с GraphQL mutations для регистрации и входа. Backend хранит постоянные пользовательские данные в MySQL и создает auth sessions, которые затем используются и GraphQL, и realtime-слоями.

На клиенте:

- auth store восстанавливает сохраненный session token из local storage
- router guard инициализирует auth до входа на защищенные маршруты
- клиент вызывает `me`, чтобы восстановить текущего пользователя

На сервере:

- GraphQL context определяет текущего пользователя по session token
- realtime connection использует ту же модель сессии для identity
- операции, которым нужна авторизация, опираются на эту session-backed identity, а не на `userId`, присланный клиентом

### 2. Initial load приложения

После авторизации клиент загружает список серверов и открывает основной authenticated layout. Страница сервера управляется выбранным `serverId` из маршрута.

Схема initial load такая:

- GraphQL для начальных постоянных данных
- LiveRail для дальнейшей синхронизации

Для выбранного сервера клиент загружает:

- метаданные сервера и структуру каналов
- текущий runtime presence snapshot

Это дает стабильную базу страницы до того, как начинают применяться live-события.

### 3. Навигационная модель

Текущий URL определяет страницу приложения и выбранный сервер. Состояние подключения к голосовому каналу не считается постоянным routing-state.

Важные следствия:

- выбранный сервер можно восстановить после перезагрузки по маршруту
- текущий голосовой канал — это runtime state в client store
- голосовое присутствие не зависит от самого факта просмотра страницы сервера
- reconnect или reload не возвращают пользователя в голосовой канал автоматически

### 4. Поток постоянных CRUD-операций

Постоянные account- и server-management операции идут через GraphQL.

Сюда относятся:

- auth
- обновление профиля
- создание сервера
- обновление сервера
- получение и перегенерация invite-ссылки
- CRUD каналов
- moderation actions
- удаление сервера

Обычный путь операции выглядит так:

1. клиент отправляет GraphQL query или mutation
2. сервер валидирует input через StrictQL и Zod-backed схемы
3. применяются доменные правила и session-backed access checks
4. Prisma читает или пишет MySQL
5. если изменилось состояние, видимое другим клиентам, сервер отправляет realtime-событие, чтобы UI обновился без reload

### 5. Поток realtime-синхронизации

Live-синхронизация идет через LiveRail contracts и runtime channels. Клиент подписывается на server-level, user-level и voice-related realtime channels. Эти подписки обновляют UI для:

- метаданных сервера
- структуры каналов
- обновлений профиля
- presence state
- изменений voice state
- screen share state
- forced disconnect сценариев

Главный приемник этих событий — client store. Realtime payload-ы нормализуются в Pinia state, а интерфейс обновляется уже от состояния store, а не напрямую от transport callback-ов.

### 6. Поток voice presence

Голосовое присутствие — live-first и runtime-only. Оно не хранится как постоянное состояние в базе данных.

Когда пользователь входит в канал или меняет канал:

1. клиент отправляет LiveRail command
2. сервер проверяет membership и доступ к каналу
3. runtime presence registry обновляет активный канал пользователя
4. сервер отправляет presence events
5. клиенты обновляют заполненность каналов и списки участников

Система гарантирует, что один пользователь может одновременно иметь только одно активное voice presence.

### 7. Поток голосового media

Media plane отделен от application plane.

- LiveRail передает voice-команды, state changes и signaling
- MediaSoup передает реальный media transport

После того как у пользователя появляется активное voice presence, клиентская mediasoup session согласовывает transports и producers/consumers через signaling bridge, построенный поверх LiveRail. Серверный media foundation управляет MediaSoup workers, routers, transports и жизненным циклом producer/consumer.

Такое разделение позволяет держать:

- application state и authorization в app/realtime слоях
- реальное распространение аудио в SFU media-layer

### 8. Mute, deafen и демонстрация экрана

Mute, deafen и screen share трактуются как server-authoritative live features.

Поток выглядит так:

1. пользователь нажимает control на клиенте
2. клиент отправляет LiveRail command
3. сервер проверяет активное presence и корректность команды
4. runtime state обновляется
5. отправляется realtime event для затронутых клиентов
6. media-поведение следует за обновленным авторитетным состоянием

Screen share живет отдельным media lifecycle относительно обычного аудио. Остальные участники того же активного канала получают демонстрацию через media/signaling flow и отображают ее в центральной области канала.

### 9. Flow приглашений и discovery

Поиск серверов и вступление используют комбинацию GraphQL и routing-а.

Текущие пути входа:

- поиск публичных серверов
- прямой переход по invite-ссылке

Для invite-ссылки:

- owner получает или перегенерирует invite link сервера
- пользователь открывает ссылку
- если он не авторизован, приложение переводит его через auth-flow
- после auth клиент продолжает invite join flow и добавляет пользователя в сервер

Для public discovery:

- авторизованные пользователи могут искать публичные серверы
- UI также может показывать рекомендованные публичные серверы
- вступление сохраняется через GraphQL и затем отражается в live-списке серверов

### 10. Модерация и forced disconnect

Модерация сервера доступна только owner. Когда owner кикает или банит участника:

1. действие сохраняется в MySQL
2. membership обновляется или блокируется
3. runtime presence очищается, если пользователь был активен в voice
4. media teardown завершается для этого пользователя
5. realtime events уведомляют все затронутые клиенты

Это не дает остаться stale UI или stale media state после moderation и держит voice presence, membership и media lifecycle согласованными.

### 11. Disconnect, refresh и reconnect

Проект отдельно обрабатывает disconnect edge cases.

Текущее поведение:

- runtime presence очищается на disconnect
- локальное клиентское voice state очищается на refresh, offline и transport disconnect
- reconnect заново поднимает realtime subscriptions и переснимает snapshots
- reconnect не возвращает пользователя в voice автоматически

Это сделано намеренно. Постоянное серверное состояние сохраняется, но активное участие в голосовом канале после disconnect или reload нужно снова инициировать вручную.

## Заметки / Важные ограничения

- Система voice-first. Текстовый чат как домен сейчас отсутствует.
- В продукте есть только роли `OWNER` и `MEMBER`.
- У сервера всегда ровно один owner, передача ownership сейчас не реализована.
- Пользователь может быть активен только в одном голосовом канале одновременно.
- Runtime presence, voice state и active screen share state — это намеренно in-memory runtime state, а не постоянные записи в базе.
- URL хранит состояние выбранного сервера, а не активного голосового подключения.
- GraphQL используется для постоянных чтений и мутаций, а LiveRail — для live-first команд и доставки realtime-событий.
- MediaSoup отвечает только за media plane и не заменяет application authorization, presence или signaling orchestration.
- Общий код в `shared/` должен оставаться dependency-light и не должен превращаться во второй runtime-пакет со framework-specific логикой.
- Текущие hard limits продукта:
  - до 2 owned servers на одного owner
  - до 5 каналов на сервер
