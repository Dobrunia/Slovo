# Дизайн проекта: voice-first платформа с серверами и каналами

## 1. Обзор

Проект — веб-приложение для голосового общения, организованное вокруг постоянных серверов и голосовых каналов. Пользователь заходит на сайт, видит список своих серверов, переходит в нужный сервер и сразу видит, в каких голосовых каналах находятся люди. Основной сценарий — быстро понять, где идет разговор, и подключиться к нему.

Продукт intentionally не является полным аналогом Discord. MVP сфокусирован на голосовом общении, видимости активных каналов и базовом управлении сервером.

В будущем может быть добавлена desktop-сборка через Tauri, но текущий приоритет — web-first реализация.

---

## 2. Цели MVP

### 2.1 Продуктовые цели

* Дать пользователю постоянные серверы с постоянными голосовыми каналами.
* Сделать максимально быстрым сценарий входа в разговор.
* Показывать в реальном времени, кто находится в каждом канале.
* Дать базовый набор управления голосом: mute, deafen, выбор устройства ввода.
* Дать базовое управление сервером одному администратору.
* Включить демонстрацию экрана уже в MVP.

### 2.2 Что не входит в MVP

* Текстовые каналы.
* Личные сообщения.
* Личные звонки.
* Временные каналы.
* Несколько администраторов на сервере.
* Передача владения сервером.
* Сложные роли и permissions matrix.
* Смена email.
* Смена пароля.
* Платные планы, подписки, биллинг.
* Мобильные приложения.

---

## 3. Основная терминология

* **User** — зарегистрированный пользователь приложения.
* **Server** — постоянное пространство/сообщество.
* **Voice Channel** — постоянный голосовой канал внутри сервера.
* **Server Member** — пользователь, состоящий в сервере.
* **Admin** — создатель сервера и единственный пользователь с административными правами на этом сервере.
* **Member** — обычный участник сервера.
* **Invite Link** — основная ссылка-приглашение на сервер.
* **Presence** — текущее присутствие пользователя в голосовом канале.
* **Audio State** — текущее состояние аудио пользователя, например mute/deafen.
* **Screen Share** — трансляция экрана пользователя внутри канала.
* **Server Ban** — запрет пользователю на доступ к серверу.

Термин «комната» не используется как официальная сущность. Официальные сущности: **server** и **voice channel**.

---

## 4. Продуктовые принципы

1. **Voice-first**: приложение строится вокруг голосового общения.
2. **Presence-first**: пользователь должен сразу видеть, где находятся люди.
3. **Минимум лишней сложности**: только нужные для MVP роли, ограничения и настройки.
4. **Постоянная структура**: серверы и каналы постоянные, а не временные.
5. **Архитектура с запасом**: решения MVP не должны блокировать дальнейшее развитие screen share, desktop-версии и расширения лимитов.

---

## 5. Функциональные требования

## 5.1 Аутентификация и аккаунт

Пользователь обязан зарегистрироваться, чтобы пользоваться системой.

### Поддерживаемые поля профиля

* email
* password
* display name
* avatar
* username

### Ограничения MVP

* email нельзя менять после регистрации
* password нельзя менять в MVP
* username нельзя менять в MVP
* display name можно менять
* avatar можно менять

### Необходимые сценарии

* регистрация
* логин
* выход из аккаунта
* просмотр собственного профиля
* изменение display name
* изменение avatar

---

## 5.2 Серверы

Пользователь может состоять в нескольких серверах и создавать собственные серверы.

### Типы серверов

* **Public server** — сервер можно найти через поиск по названию и вступить в него.
* **Private server** — сервер недоступен через публичный поиск, вступление только по приглашению.

### Ограничения MVP

* один пользователь может создать максимум **2 сервера**
* на каждом сервере может быть максимум **5 каналов**
* у сервера есть только **один admin**
* admin — это создатель сервера
* передача admin-прав невозможна
* назначение дополнительных admin невозможно

### Действия пользователя

* создать сервер, если не достигнут лимит
* открыть сервер из списка своих серверов
* вступить в public server через поиск
* вступить в server по invite link
* покинуть сервер, если пользователь не admin

### Действия admin

* изменить название сервера
* изменить avatar сервера
* удалить сервер
* выгнать участника
* забанить участника
* перевыпустить invite link

---

## 5.3 Invite links

У каждого сервера есть одна основная invite link.

### Правила

* ссылка существует как постоянная основная ссылка сервера
* лимита по времени нет
* лимита по использованию нет
* admin может перевыпустить ссылку
* после перевыпуска старая ссылка становится недействительной

### Поведение для server type

* для private server ссылка — основной способ вступления
* для public server ссылка также существует как удобный способ прямого входа

---

## 5.4 Поиск серверов

В системе есть поиск публичных серверов по названию.

### Требования

* пользователь может искать public server по названию
* в результатах поиска показываются только public servers
* private servers не отображаются в поиске
* пользователь может открыть карточку public server и вступить в него

Для MVP поиск может быть простым, без сложного ранжирования и категорий.

---

## 5.5 Каналы

Каналы внутри сервера являются только голосовыми и только постоянными.

### Ограничения

* на сервере максимум 5 каналов
* каналы не бывают текстовыми
* каналы не бывают временными
* личных звонков нет

### Действия member

* открыть список каналов сервера
* войти в канал
* выйти из канала
* перейти из одного канала в другой

### Действия admin

* создать канал
* переименовать канал
* удалить канал
* изменить порядок каналов

### Правила присутствия

* один пользователь может находиться максимум в **одном голосовом канале одновременно**
* при переходе в другой канал пользователь автоматически покидает предыдущий
* пользователь, не находящийся в канале, не отображается как активный участник канала

---

## 5.6 Presence и отображение активности

Presence является ключевой продуктовой функцией.

### Что должно быть видно пользователю

* список каналов на сервере
* количество пользователей в каждом канале
* список пользователей внутри каждого канала

### Что не нужно показывать в MVP

* глобальный online/offline статус всех участников сервера
* статус пользователей вне каналов
* last seen и подобные индикаторы

### Ключевое UX-правило

Интерфейс сервера должен отвечать на вопрос:
**«Кто сейчас находится в каком голосовом канале?»**

Именно это состояние является главным real-time представлением сервера.

---

## 5.7 Голосовые функции

### Для каждого пользователя

* подключение к голосовому каналу
* отключение от голосового канала
* mute/unmute микрофона
* deafen/undeafen (не слышать других)
* выбор устройства ввода (microphone input device)

### Поведение deafen

Когда пользователь включает deafen:

* он перестает слышать других пользователей
* он не может говорить

Это поведение должно быть аналогично Discord.

### Базовые ожидания от качества

* быстрое подключение к голосу
* низкая задержка
* стабильная работа при каналах порядка 10–20 участников

---

## 5.8 Демонстрация экрана

Screen share входит в MVP.

### Требования MVP

* пользователь в голосовом канале может начать демонстрацию экрана
* пользователь может остановить демонстрацию экрана
* другие участники канала могут видеть активную демонстрацию
* одновременно у одного пользователя может быть не более одной активной демонстрации

### Уточнение для MVP

В первой версии допустимо ограничиться базовым сценарием без сложных режимов:

* без нескольких параллельных демонстраций от одного пользователя
* без приоритизации качества
* без расширенных настроек битрейта
* без записи

Screen share должен быть заложен в media-архитектуру как полноценная функция MVP.

---

## 5.9 Модерация

Модерация для MVP минимальна и завязана только на admin.

### Доступные действия admin

* выгнать участника из сервера
* забанить участника на сервере

### Бан

Бан действует независимо от текущего статуса пользователя.

Admin может забанить:

* пользователя, который онлайн
* пользователя, который оффлайн
* пользователя, который находится в голосовом канале

### Поведение при бане

* пользователь удаляется из сервера
* если пользователь в этот момент находится в канале, он немедленно отключается от канала
* пользователь теряет доступ к серверу
* пользователь не может повторно вступить в сервер, пока бан активен

---

## 6. User flows

## 6.1 Регистрация и первый вход

1. Пользователь открывает сайт.
2. Регистрируется с email, password, username и display name.
3. Попадает в основной интерфейс.
4. Видит пустой список серверов или уже доступные серверы.

## 6.2 Создание сервера

1. Пользователь нажимает «создать сервер».
2. Система проверяет лимит: не более 2 серверов на создателя.
3. Пользователь задает название сервера.
4. Сервер создается, пользователь становится admin.
5. По умолчанию создается invite link.
6. Admin может создать до 5 голосовых каналов.

## 6.3 Вступление в сервер по поиску

1. Пользователь открывает поиск.
2. Вводит название сервера.
3. Видит только public servers.
4. Выбирает сервер.
5. Вступает в него как member.

## 6.4 Вступление по invite link

1. Пользователь открывает invite link.
2. Система валидирует ссылку.
3. Если сервер private или public — пользователь может вступить, если не забанен.
4. После вступления сервер появляется в списке серверов пользователя.

## 6.5 Подключение к голосовому каналу

1. Пользователь открывает сервер.
2. Видит список каналов и участников в каждом канале.
3. Выбирает канал.
4. Подключается к нему.
5. Остальные пользователи сервера видят его в составе этого канала.

## 6.6 Переход между каналами

1. Пользователь находится в канале A.
2. Нажимает на канал B.
3. Система отключает его от канала A.
4. Система подключает его к каналу B.
5. Presence обновляется для всех участников сервера.

## 6.7 Screen share

1. Пользователь находится в голосовом канале.
2. Нажимает «начать демонстрацию».
3. Браузер предлагает выбрать экран/окно.
4. После подтверждения начинается трансляция.
5. Другие участники канала видят активный screen share.
6. Пользователь может остановить трансляцию вручную или она завершается при завершении browser share session.

## 6.8 Бан пользователя

1. Admin выбирает участника сервера.
2. Нажимает «ban».
3. Система создает запись бана.
4. Если пользователь был в канале, он немедленно отключается.
5. Сервер удаляется из списка серверов этого пользователя.
6. Повторный вход в сервер запрещен.

---

## 7. Нефункциональные требования

## 7.1 Производительность

* Реал-тайм обновление состава каналов должно происходить почти мгновенно.
* Задержка presence-обновлений должна быть минимальной.
* Канал должен быть рассчитан на ориентировочно 10–20 участников.

## 7.2 Надежность

* При разрыве соединения состояние пользователя должно корректно очищаться.
* При закрытии вкладки пользователь должен исчезать из канала.
* При восстановлении соединения приложение должно уметь восстановить состояние клиента.

## 7.3 Масштабируемость

* Ограничения MVP задаются как конфигурируемые server-side константы.
* Лимиты должны быть реализованы так, чтобы их можно было изменить без переработки core-модели.

Примеры лимитов:

* `MAX_SERVERS_PER_OWNER = 2`
* `MAX_CHANNELS_PER_SERVER = 5`

## 7.4 Безопасность

* Приватные серверы не должны быть доступны через поиск.
* Invite links должны уметь инвалидироваться перевыпуском.
* Забаненный пользователь не должен иметь возможность повторно вступить в сервер по старой или новой ссылке.
* Все административные действия должны валидироваться на backend.

## 8. Архитектурная модель

## 8.1 Высокоуровневая схема

### WebSocket

Используется для:

* аутентифицированной real-time сессии
* signal/events обмена
* presence-обновлений
* изменений состава каналов
* служебных событий сервера
* сигналинга для WebRTC

### WebRTC

Используется для:

* голосовых потоков
* screen share потоков

### Backend

Отвечает за:

* аутентификацию
* авторизацию
* хранение серверов, каналов, участников, банов и invite links
* проверку лимитов
* поддержку поиска public servers
* координацию presence и сигналинга

### Frontend

Отвечает за:

* интерфейс серверов и каналов
* отображение участников каналов
* управление mute/deafen
* выбор microphone input device
* запуск и остановку screen share
* отображение active media state

---

## 8.2 Почему нужен careful media design

Так как в MVP одновременно присутствуют:

* голосовые каналы до 10–20 человек
* screen share

архитектура media-подсистемы не должна строиться как наивное решение, которое годится только для маленьких peer-to-peer звонков.

В design phase следует учитывать, что voice + screen share при 10–20 участниках создают заметную нагрузку, и media-routing должен быть спроектирован с запасом.

На уровне дизайн-дока фиксируется принцип:
**MVP-реализация не должна блокировать последующий переход к более масштабируемой media-топологии.**

* is_deafened

## 10. Ограничения и бизнес-правила

1. Пользователь обязан иметь аккаунт.
2. Пользователь может создать не более 2 серверов.
3. На сервере может быть не более 5 каналов.
4. На сервере может быть только один admin.
5. Admin — создатель сервера.
6. Admin-права нельзя передать.
7. Один пользователь может быть только в одном голосовом канале одновременно.
8. Каналы только голосовые.
9. Каналы только постоянные.
10. Invite link у сервера одна активная.
11. Invite link может быть перевыпущена admin без лимитов и сроков.
12. Private servers не должны отображаться в поиске.
13. Public servers должны находиться через поиск по названию.
14. Email, password и username в MVP не редактируются.
15. Display name и avatar редактируются.
16. Забаненный пользователь не может вступить в сервер повторно.
17. Бан должен работать и для онлайн, и для оффлайн пользователей.
18. При бане пользователя из активного канала он немедленно отключается.

## 11. UI/UX-ожидания

## 11.1 Основной layout

Базовый экран должен содержать:

* список серверов
* выбранный сервер
* список голосовых каналов
* список участников внутри каждого канала
* нижнюю/боковую область с собственным состоянием пользователя

## 11.2 Основные UX-акценты

* пользователь сразу видит активность по каналам
* вход в канал должен занимать минимум действий
* mute/deafen должны быть доступны в один клик
* визуально должно быть понятно, кто сейчас говорит, если voice activity будет добавлена позже
* screen share должен запускаться из канала естественным способом

## 11.3 Отдельные UX-принципы

* не перегружать интерфейс функциями, которых нет в MVP
* не показывать лишние статусы вне каналов
* основное внимание — на каналах и их участниках

## 12. Ошибки и edge cases

### Аккаунт

* попытка зарегистрироваться на уже занятый email
* попытка войти с неверным паролем

### Сервер

* попытка создать третий сервер
* попытка вступить в сервер при активном бане
* попытка открыть удаленный сервер

### Каналы

* попытка создать шестой канал
* попытка подключиться к удаленному каналу
* корректный auto-leave из старого канала при переходе в новый

### Invite links

* попытка использовать перевыпущенную старую ссылку
* попытка использовать несуществующую ссылку

### Presence

* закрытие вкладки
* разрыв WebSocket-соединения
* зависшая сессия пользователя
* повторное подключение после reconnect

### Screen share

* пользователь отменил выбор экрана
* браузер завершил screen share извне
* пользователь вышел из канала во время демонстрации
* admin забанил пользователя во время screen share

---

## 13. Конфигурируемые лимиты

Все лимиты MVP должны быть server-side конфигурируемыми константами.

Примеры:

* max servers per owner = 2
* max channels per server = 5

Важно: даже если в будущем появятся тарифы или расширения, в MVP эти лимиты просто проверяются на backend и не требуют отдельной платежной логики.

## 15. Итоговая формулировка MVP

MVP — это web-first платформа голосового общения с постоянными серверами и постоянными голосовыми каналами, где пользователь видит, кто находится в каждом канале, может быстро подключаться к разговору, управлять своим аудио, а также использовать демонстрацию экрана. Управление сервером максимально упрощено: один admin, простая модель public/private, одна invite link, базовый поиск публичных серверов и жесткие лимиты на количество серверов и каналов.

Стек
Фронтенд

Vue 3 — основной UI-фреймворк приложения. Официальная документация Vue рекомендует современный setup и хорошо сочетается с Vite.

Vite — dev server и production build для клиентской части.

Vitest — тестирование клиентской части и общего TypeScript-кода рядом с фронтендом.

DobruniaUI Vue (dobruniaui-vue) — основная UI-библиотека (готовые Vue 3-компоненты, utility-классы, design tokens и темы, чтобы минимизировать объем собственных scoped-стилей). В README указаны plugin-режим через app.use, named imports, styles.css и tree shaking.

LiveRail Client (@liverail/client, при необходимости @liverail/client/socket-io) — клиентский real-time runtime (подписки, reconnect flow, event appliers и typed real-time взаимодействие).

LiveRail Contracts (@liverail/contracts) — общий контрактный слой, который используется и на фронте тоже (команды, события, валидация, ошибки и registry primitives, общие для клиента и сервера).

MediaSoup client-side integration — клиентская часть media-подключения для голоса и screen share через WebRTC-транспорт, управляемый серверной MediaSoup-инфраструктурой. Это и будет реальный media-слой приложения.

Бэкенд

Node.js + TypeScript — основа серверной части; и StrictQL, и LiveRail описаны как TypeScript-first библиотеки для Node.js.

GraphQL Yoga (graphql-yoga) — GraphQL server (HTTP GraphQL runtime, GraphiQL, spec-compliant GraphQL over HTTP, удобная серверная оболочка для схемы и контекста). Официально Yoga описывается как batteries-included cross-platform GraphQL server.

StrictQL (strictql) — GraphQL contract layer (contract-driven/code-first GraphQL для Node.js + TypeScript с обязательной input/output валидацией через Zod, predictable error shape, auth policies и security limits). В README отдельно показана интеграция с Yoga.

LiveRail Server (@liverail/server, при необходимости @liverail/server/socket-io) — серверный real-time runtime (policies, lifecycle hooks, transport adapters и typed event/command processing для presence, signaling и server events).

LiveRail Contracts (@liverail/contracts) — общий слой контрактов между backend и frontend.

MediaSoup — media backend (SFU для голосовых каналов и демонстрации экрана, работающий поверх WebRTC и лучше подходящий для многопользовательских voice-комнат, чем наивная peer-to-peer схема). Это логичное ядро media-части для вашего MVP с voice + screen share.

Если оформить это как уже готовый абзац для дизайн-дока, я бы написал так:

Технологический стек проекта: клиентская часть строится на Vue 3, Vite и Vitest, а интерфейс собирается преимущественно на DobruniaUI Vue (готовая Vue 3 UI-библиотека с компонентами, utility-классами, design tokens и темами для минимизации собственного scoped CSS). Для real-time слоя на клиенте используется LiveRail Client (typed client runtime с подписками, reconnect flow и event appliers), а общий контракт между клиентом и сервером описывается через LiveRail Contracts (команды, события, валидация, ошибки и registry primitives). Media-часть приложения реализуется через MediaSoup, который управляет голосовыми потоками и демонстрацией экрана поверх WebRTC. Серверная часть строится на Node.js + TypeScript, использует GraphQL Yoga (GraphQL server runtime) и StrictQL (contract-driven GraphQL-слой с обязательной input/output валидацией, auth policies, unified errors и готовой интеграцией с Yoga). Для real-time backend используется LiveRail Server (серверный runtime для команд, событий, presence и signaling), а LiveRail Contracts выступает общим типизированным протоколом между фронтендом и бэкендом.
